## §2 Multi-perspective review

### Code review (general-purpose agent)

Verdict: **APPROVE WITH NOTES**. No CRITICAL / HIGH. Two MEDIUMs surfaced and both **resolved in this run**:

- **MEDIUM-1 (resolved)**: `req.url.includes('api-version=')` was a loose substring check — would have false-matched paths containing `x-api-version=` or the literal as a path segment. Tightened to regex `/[?&]api-version=/` so only real query params count.
- **MEDIUM-2 (resolved)**: CONFIGURATION.md directory carried both a `microsoft` row and a new `azure` row pointing at the same vendor. Added an explicit disambiguation callout — `azure` is the modern adapter, `microsoft` is a legacy back-compat alias.

### Security review (general-purpose agent)

Verdict: **APPROVE**. No CRITICAL / HIGH / MEDIUM. Six LOW findings, all favourable:

- `encodeURIComponent` neutralises any `?`/`&`/`#`/`=`/`"`/`<`/`>` in the env value → no query-string injection.
- `req.url` mutation is path-only — http-proxy `target` is bound to `config.upstreamUrl` at module load. **No SSRF surface.**
- `azureApiVersionParam` is a module-level const computed once at import — immutable, no cross-request race.
- No header logging anywhere; `requestLogger.js` does not read `req.headers`; `api-key` is never logged, stored, or echoed.
- Setup-tab snippet is a pure string template — no real secret ever reaches the DOM.
- Substring loose-match noted (now resolved above).

## §4b Dashboard polish — y-axis precision fix

While the harness was running, the user reported the Tokens chart showing
y-axis labels like `0.6000000000000001` / `0.39999999999999991` — IEEE-754 float
artifacts from Chart.js's default tick stringification. The old callback was
`(v) => Math.abs(v)` which returned the number unmasked.

Fix in `public/app.js`: new module-level `fmtAxis(v)` helper applied to both
`makeLineChart` (RPS / p95) and `makeDivergingTokensChart`. Magnitude-driven
precision: ≥100 integer, ≥10 integer, ≥1 → 1 decimal, ≥0.1 → 2 decimals, else
2 significant figures (so 0.0006 stays "0.0006" instead of collapsing to "0.00").

Verified live in the running container (file pushed via `docker cp`, since
the production `docker-compose.yml` does not mount `./public` live — only the
auto-loaded `docker-compose.override.yml` does, and the E2E suite explicitly
uses `-f docker-compose.yml` so the override is suppressed). All three charts
now render clean ticks: RPS axis `0 / 0.1 / 0.2 / … / 1.0`, p95 axis
`0 / 200 / 400 / … / 1600`, Tokens axis `-20 / -15 / -10 / -5.0 / 0 / 5.0 / …`
with the `(v) => fmtAxis(v)` callback applied to the (signed) raw tick values.

## §4b.1 Logs tab — backfill of proxied-request history

After §4a the user noted the Logs tab was empty of the 15 real Mistral calls
they had just fired. Investigation showed `loadLive()` in `public/app.js`
fetched only `/api/logs` (the file-backed app log) on initial render. By
design the proxy hot path never writes to that file — proxied request events
live in the metrics ring buffer and stream over `/api/metrics/stream`. Live
SSE events did reach the Logs panel via the existing `request` handler in
`connectMetricsSSE`, but historical proxy events fired before page load were
unreachable.

Fix in `public/app.js:loadLive()` (frontend-only — no server change, hot-path
invariant intact):

- Fetch `/api/logs?limit=500` AND `/api/metrics/recent?limit=500` in parallel.
- Merge into one array tagged with `type: 'proxy' | 'internal' | 'system'`.
- Sort newest-first by `ts`. This also corrects a pre-existing ordering bug —
  the previous code pushed into the buffer oldest-first, which put oldest at
  the top of the DOM, while the live `bufferAndRender` path always prepends
  newest. Both paths now agree on newest-at-top.

Re-fired `scripts/e2e-real-prompts.py` (15/15 OK on the second pass — rate
limit had cleared). Re-loaded the Logs tab. All 15 fresh `POST /proxy/v1/chat/completions`
rows visible at the top with `200`, `mistral-small-latest`, real
`in↓ out↑ tok` counts and real `$0.00xxxx` costs. The previous burst's traffic
appears further down the list, properly time-ordered. Filter checkboxes for
proxy / internal / system continue to gate visibility correctly.

## §4b.2 Logs tab — timestamp format

Two follow-on fixes after §4b.1 went in:

1. **Format consistency.** App-log rows (internal/system) were rendering the
   raw ISO timestamp (`2026-05-12T13:06:59.799Z`), while proxy rows used
   `fmtTime(ev.ts)` (`13:06:59`). The split made the panel hard to scan.
   `renderLogRow` for non-proxy items now also pipes through `fmtTime`.
2. **Full precision in browser TZ.** User asked for the full date + millisecond
   precision in the browser's timezone. `fmtTime` upgraded to emit
   `YYYY-MM-DD HH:MM:SS.mmm` via native `Date` getters (which already return
   browser-local values), so no external library is needed. The same helper
   is shared by the recent-requests and top-cost tables, so the new format
   propagates everywhere a timestamp is rendered.

Verified live: all Logs rows now render identically, e.g. `2026-05-12 15:05:50.425`
for a proxy `200` and `2026-05-12 15:09:49.903` for an internal `GET /api/logs`.
Sortable as a string, copy-pasteable as an ISO-ish local time, ms-precise.

## §6 Azure synthetic E2E (no live key)

The vitest integration test `tests/proxy/azure-api-version.test.js` runs the **same code path as production** (full `createApp()` + a real `http.Server` stub upstream) under `PROXY_PROVIDER=azure`. Three scenarios validated end-to-end at the wire level:

1. **Append**: client sends `/proxy/openai/deployments/gpt-4o-mini/chat/completions` (no api-version) → upstream receives the path with `?api-version=2024-10-21` appended.
2. **Preserve**: client sends `…?api-version=2025-01-01` → upstream receives `?api-version=2025-01-01` exactly once; no double-append, default does **not** override caller intent.
3. **Header forwarding**: client sends `api-key: sekret-123` → upstream receives the same header byte-for-byte.

Plus a fourth scenario in a separate `describe`: with `PROXY_PROVIDER=openai` the hook is dormant — upstream sees no `api-version=` even though the env var is set.

All four pass. **Live-Azure E2E is deferred** — gated on the user supplying an Azure OpenAI resource + key in a future session, and is the only outstanding gate before v0.2.7 can ship.
