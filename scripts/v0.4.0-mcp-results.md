## Operator notes for the runbook

These observations should be folded into a future revision of
[`scripts/v0.4.0-mcp-scenarios.md`](v0.4.0-mcp-scenarios.md):

1. **Section 2.8 wording.** The runbook's literal "stacktrace-dedupe fires"
   assertion is too strict in a pipeline context — `repeat-line-dedupe`
   sits earlier in the registry order (`registry.js:16-27`) and will
   collapse perfectly identical frames first. Either of the two
   compressors firing satisfies the user-visible "frames got collapsed"
   intent. The harness assertion was loosened accordingly.
2. **`X-Guardrails: off` bypass events.** When a request bypasses via the
   header, the middleware returns `next()` before recording any metric.
   The dashboard `requestsBypassed` counter therefore stayed at 0 even
   though we sent bypass headers twice. This matches Compactor's behavior
   (bypass-by-header = silent passthrough). If we want bypass visibility
   on the dashboard later, we'd need a small metrics tweak — capture as
   `v0.4.1` follow-up.
3. **`COMPACTOR_TOOL_RESULT_ONLY` for the runbook.** The default `true`
   limits mutation to `tool_result` / `role:tool` blocks. The runbook
   sends bloated payloads as a top-level user message, so the operator
   must set `COMPACTOR_TOOL_RESULT_ONLY=false` for the run (now reflected
   in the runbook's `.env` template) — or wrap payloads as `tool` role.
4. **`UPSTREAM_URL` for Mistral.** Must be `https://api.mistral.ai`
   (no trailing `/v1`). The proxy preserves the request path verbatim
   after stripping the prefix, so `/proxy/v1/chat/completions` correctly
   becomes `https://api.mistral.ai/v1/chat/completions`. The runbook's
   `.env` template already has this right; flagging for the operator.
